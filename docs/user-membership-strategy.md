### 3.3 updatedAt 更新时机

以下操作会更新计数器的 `updatedAt` 字段：

| 操作 | 更新 updatedAt | 设置 hasUnsyncChanges |
|------|---------------|----------------------|
| 计数 +1/-1 | ✅ | ✅ |
| 修改名称 | ✅ | ✅ |
| 修改目标行数 | ✅ | ✅ |
| 修改备忘录 | ✅ | ✅ |
| 计时器开始/暂停/重置 | ✅ | ✅ |
| 计时器自动累计 | ❌ | ❌ |

`updatedAt` 用于同步时判断数据新旧，决定保留本地还是云端数据。

> **说明**：计时器自动累计是后台运行行为，不触发同步，避免频繁请求。计时数据在下次同步时一并保存。

## 五、数据同步时机

采用精简同步策略，在关键节点同步数据，避免频繁请求带来的成本与复杂度：

| 触发场景       | 同步策略                   | 说明                                     |
| -------------- | -------------------------- | ---------------------------------------- |
| 创建/删除计数器 | 实时同步（仅已登录用户）   | 重要操作，用户期望立即生效               |
| 小程序切后台   | 立即同步                   | 保证数据不丢失                           |
| 小程序打开     | 拉取云端数据并合并         | 换设备后能拿到最新数据                   |
| 登录/退出登录   | 立即同步                   | 确保登录状态变更时数据一致               |
| 定时心跳       | 每10分钟检查（有变更时）   | 兜底机制，防止异常退出导致数据丢失       |

> **说明**：计数变更（加/减操作）不同步到云端。针织用户频繁操作计数器，实时同步会增加成本和复杂度。通过切后台、心跳等时机兜底，数据丢失风险可控。

**小程序打开时的处理流程：**

```
小程序打开
    │
    ▼
是否已登录？
    │
    ├─ 否 → 无操作
    │
    └─ 是 → 拉取云端数据
            │
            ▼
         本地是否有未同步变更（hasUnsyncChanges）？
            │
            ├─ 否 → 云端数据覆盖本地
            │
            └─ 是 → 执行合并流程（同 7.2）
```

**实现要点：**
- 每个计数器数据增加 `updatedAt` 字段记录最后修改时间
- 本地存储 `lastSyncTime` 记录上次同步时间
- 设置 `hasUnsyncChanges` 标记：
  - 用户主动操作（计数、改名、改备忘录、计时器开始/暂停/重置）设为 `true`
  - 计时器自动累计**不触发**（避免频繁同步）
  - 同步成功后设为 `false`
  - 定时心跳仅在 `hasUnsyncChanges === true` 时触发同步，避免无效请求

### 5.1 同步失败处理

| 场景 | 处理策略 |
|------|----------|
| 创建计数器同步失败 | 本地创建成功，下次打开小程序时重试 |
| 删除计数器同步失败 | 本地删除成功，几秒后自动重试（最多3次），仍失败则下次打开小程序时重试 |
| 切后台同步失败 | 用户无感知，下次打开小程序时补救 |

### 5.2 离线使用规则

| 场景 | 处理方式 |
|------|----------|
| 已登录用户离线操作 | 正常使用，上线后自动同步 |
| 离线创建计数器 | 根据本地缓存的会员状态判断上限 |
| 离线删除计数器 | 本地删除，上线后同步删除到云端（墓碑机制） |

**离线上限检查规则：**

离线时无法获取云端数据，使用本地缓存的会员状态判断：

| 用户类型 | 离线创建计数器限制 |
|----------|-------------------|
| 会员（本地缓存） | 无上限限制 |
| 非会员/未登录 | 本地已有 ≥3 个则禁止创建 |

> **说明**：会员状态在登录时缓存到本地，离线时使用缓存判断。

上线后合并时，如果总数超过 3 个（非会员），按"老用户超限"规则处理（保留但禁止新增）。

### 5.3 数据留存

| 数据位置 | 留存策略 |
|----------|----------|
| 本地数据（未登录） | 由用户自己管理，应用不主动清理 |
| 云端数据（已登录） | 长期保留 |

## 六、登录时的数据处理

### 6.1 未登录用户首次登录

当未登录用户首次登录时（本地无登录记录），系统自动将本地计数器上传并与云端数据合并。

**合并原则：**

- 数据不删，只限新增
- Key 冲突时，重命名本地 key，两边都保留（不合并数据）

> **说明**：Key 由时间戳或随机数生成，冲突概率极低。简单处理即可。

**合并流程：**

```
用户登录
    │
    ▼
拉取云端数据
    │
    ▼
遍历本地计数器：
    - 云端无此 key → 上传到云端
    - 云端有此 key → 重命名本地 key（如 counter_abc_2），上传到云端
    │
    ▼
遍历云端计数器：
    - 本地无此 key → 下载到本地
    │
    ▼
合并完成 → Toast 提示
```

### 6.2 已登录用户重新登录

用户退出后重新登录同一账号时，执行数据合并。

**合并流程：**

同 7.2 重新登录数据合并。

### 6.3 不同账号登录

当用户A退出后，用户B在同一设备登录时：

```
退出用户A：
    1. 同步A的本地数据到云端
    2. 清除本地登录状态
    3. 保留本地数据（不删除计数器）

登录用户B：
    1. 清空本地计数器数据
    2. 拉取B的云端数据到本地
```

**设计意图：**

- 退出时保留本地数据，方便用户继续使用未登录功能
- 登录新账号时清空本地，保证每个账号的数据不混淆

## 七、退出登录与数据合并

### 7.1 退出登录处理

**流程：**

```
用户点击退出登录
    │
    ▼
同步当前账号的本地数据到云端
    │
    ▼
清除本地登录状态
    │
    ▼
保留本地数据（不删除）
```

| 数据位置 | 处理方式 |
| -------- | -------- |
| 本地数据 | 保留（不清除） |
| 云端数据 | 保留 |

> **说明**：退出时先同步，确保该账号的数据完整保存到云端。本地数据保留，方便用户重新登录或切换账号时使用。

### 7.2 重新登录数据合并

采用时间戳合并策略，并使用墓碑机制处理删除操作：

```
重新登录时：
1. 拉取云端数据
2. 对比本地数据时间戳 vs 云端数据时间戳
3. 对于每个计数器：
   - 本地有、云端无 → 上传到云端
   - 本地无、云端有（正常）→ 下载到本地
   - 本地无、云端有（墓碑）→ 忽略（已删除）
   - 两边都有（正常）→ 取最新时间戳的数据
   - 两边都有（云端墓碑）→ 比较 updatedAt 与 deletedAt
```

| 数据类型       | 合并策略                         |
| -------------- | -------------------------------- |
| 计数器列表     | 取并集（合并所有计数器）         |
| 单个计数器数据 | 按 `updatedAt` 时间戳，取最新的  |
| 图解数据       | 按 `updatedAt` 时间戳，取最新的  |

### 7.3 删除处理（墓碑机制）

**为什么用墓碑机制：**

直接删除云端数据会导致其他设备无法感知删除操作。墓碑机制保留删除记录，让其他设备在同步时正确处理。

**数据结构：**

```javascript
// 云端计数器数据
{
  "counter_abc": {
    name: "毛衣",
    currentCount: 50,
    targetCount: 100,
    // ...
    updatedAt: Date,      // 最后修改时间
    deletedAt: Date | null  // 删除时间（null 表示未删除）
  }
}
```

**删除操作流程：**

用户删除计数器时，不直接删除云端记录，而是设置 `deletedAt` 字段：

1. 本地：删除该计数器
2. 云端：设置 `deletedAt = 当前服务器时间`，保留其他数据

**同步时的墓碑处理：**

| 场景 | 条件 | 处理方式 |
|------|------|----------|
| 云端墓碑，本地无 | - | 忽略，不下载 |
| 云端墓碑，本地有 | 本地 updatedAt < 云端 deletedAt | 删除本地（删除后本地无修改） |
| 云端墓碑，本地有 | 本地 updatedAt > 云端 deletedAt | 保留本地，上传云端（删除后又有修改，"复活"） |

**墓碑清理：**

超过 30 天的墓碑数据可由云函数定时任务清理，真正删除记录。

### 7.4 其他边界情况

| 场景                         | 处理方式                         |
| ---------------------------- | -------------------------------- |
| 退出后本地新增计数器         | 重新登录时上传到云端             |
| 退出后本地修改计数           | 按时间戳合并，本地更新则保留本地 |
| 超限数据（如>3个计数器）     | 老用户保留；新用户按当前限制处理 |
